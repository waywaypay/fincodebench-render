FinCodeBench

A coding-agent eval suite for financial data tasks

FinCodeBench measures how well a coding agent extracts structured financial data, writes correct financial code, runs multi-step agentic computations, and debugs broken models — the work done by quantitative analysts, IR teams, and fintech developers. It is the domain where model errors are most costly: an off-by-one in a DCF model or a hallucinated revenue figure is a wrong investment decision, not an inconvenience.

Scoring methods

Four scorers are fully deterministic; one uses a separate Claude call as judge. Every task is reduced to a single normalized score in [0, 1].

exact_json

Parse the first JSON object/array in the response and compare it to the expected output by deep equality. Score is 1.0 on an exact match, else 0.0. Used for extraction tasks with unambiguous expected outputs.

fuzzy_number

Extract every number from the response; score 1.0 if any one falls within tolerance (default ±2%) of the expected value, else 0.0. Tolerant of formatting differences in single-number computations.

fuzzy_dict

Parse a JSON dict and compare each key's value to the expected within tolerance (default ±0.5%). Score is the fraction of keys matched (0–1) — partial credit for multi-field outputs such as margin-by-quarter.

functional

Extract the ```python block, append a hidden unit test suite, and execute it in a sandboxed subprocess (30s timeout). Score is 1.0 only if the process exits 0 and prints PASS, else 0.0. The most rigorous method.

llm_judge

A separate Claude call (the judge model, deliberately distinct from the runner to avoid self-grading) scores the response 0–4 against a per-task rubric with forced-JSON output, normalized to 0–1 (score / 4). Used for agentic tasks where output format varies and execution matters.

pass@0.75

The headline rate. A task "passes" when its normalized score is ≥ 0.75. Reported overall and per category alongside the mean score.

How a single task is run

The runner loop

The agent receives the task prompt (plus any context data) and exactly one tool: execute_python, which runs Python in a sandboxed subprocess (30s timeout) and returns stdout/stderr. It loops up to the task's max_turns, calling the tool as needed. Every turn, tool call, and output is captured as a full trajectory. max_tokens = 4096 per turn.

Runner model is configurable via FINCODEBENCH_MODEL (default claude-haiku-4-5).

The judge rubric (0–4)

4Perfect — correct, complete, well-structured
3Good — mostly correct, minor issues
2Partial — right direction, key errors
1Poor — attempts but substantially wrong
0Fail — wrong, crashed, or refused

Code that crashes or returns wrong output is capped at 1; a refusal scores 0. The judge scores strictly against the rubric, not its own opinion.

Trigger a new run
