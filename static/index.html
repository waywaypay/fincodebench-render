FinCodeBench

A coding-agent eval suite for financial data tasks

FinCodeBench measures how well a coding agent does the core work of financial professionals: pulling structured data out of financial tables, filings, press releases, transcripts, and presentations; writing correct financial code from scratch; running fixed multi-step workflows — writing code, running it, reading the output, and refining until the numbers come out right; working agentically on open-ended problems — discovering data across several tools, deciding an approach, and reconciling messy inputs; and fixing bugs in broken financial models like valuation, forecasting, and ratio code.

Task categories

Every task belongs to one of seven categories. The category determines what is tested; each task's scoring method is then chosen to fit the shape of its correct answer (see below). Counts are computed live from the loaded task set.

Scoring methods

Four scorers are fully deterministic; one uses a separate Claude call as judge. Each task is assigned the single method that fits the shape of its correct answer — an exact value, one number, a multi-field result, runnable code, or an open-ended analysis — and is reduced to a normalized score in [0, 1].

exact_json

Parses the first JSON object/array in the response and compares it to the expected output by deep equality — 1.0 on an exact match, else 0.0. Why: used when the answer is a single discrete, unambiguous value or record (most extraction tasks); there is exactly one correct output, so anything but an exact match is wrong.

fuzzy_number

Extracts every number in the response and scores 1.0 if any falls within tolerance (default ±2%) of the expected value, else 0.0. Why: used for computations that yield one number (a ratio, WACC, an Altman Z-score); the tolerance forgives rounding and formatting without rewarding a wrong figure.

fuzzy_dict

Parses a JSON dict and compares each key's value within tolerance (default ±0.5%); the score is the fraction of keys matched. Why: used when the answer is several related numbers (e.g. margin-by-quarter) — partial credit rewards getting most of a multi-field result right.

functional

Extracts the ```python block, appends a hidden unit-test suite, and runs it in a sandboxed subprocess (30s timeout) — 1.0 only if the process exits 0 and prints PASS, else 0.0. Why: used for code_generation and debug, where the deliverable is code; correctness is settled by executing it, not by matching text.

llm_judge

A separate Claude call (the judge model, deliberately distinct from the runner to avoid self-grading) scores the response 0–4 against a per-task rubric, normalized to score / 4. Why: used for workflow and agentic tasks, whose output mixes code, execution and prose with no single canonical answer — only a rubric can judge whether the analysis was done well.

pass@0.75

≥ 0.75

The headline rate. Every task is reduced to a normalized score in [0, 1]. If that score hits 0.75 or above, the task counts as passed. pass@0.75 is simply the share of tasks that passed — so a result of 46% means the model cleared the bar on 46% of tasks and fell short on the other 54%. Reported overall and per category alongside the mean score.

How a single task is run

The runner loop

The agent receives the task prompt (plus any context data) and at least one tool: execute_python, which runs Python in a sandboxed subprocess (30s timeout) and returns stdout/stderr. Agentic tasks add data tools — list_files / read_file over a private working directory and fetch_filing over a filings service whose data never touches disk — so the agent must choose which tool to call, and when, based on what it finds. It loops up to the task's max_turns; every turn, tool call, and output is captured as a full trajectory. max_tokens = 4096 per turn.

Runner model is configurable via FINCODEBENCH_MODEL (default claude-haiku-4-5).
